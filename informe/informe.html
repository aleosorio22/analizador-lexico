Análisis del Problema

¿Qué se pide?

El enunciado solicita construir un Analizador Léxico: un programa capaz de leer código fuente escrito en cualquier lenguaje de alto nivel e identificar y clasificar cada elemento del texto en categorías llamadas tokens.

Un analizador léxico es la primera fase de un compilador. Antes de que el compilador pueda entender la estructura o el significado de un programa, necesita saber qué tipo de "palabras" lo componen: si una secuencia de caracteres es un número, un nombre de variable, un operador, una palabra reservada, etc.

Elementos a reconocer

El enunciado define 8 categorías de tokens que el programa debe ser capaz de identificar:

  1. Identificadores — nombres de variables o funciones, hasta 10 caracteres.
  2. Números enteros sin signo — solo valores entre 0 y 100.
  3. Cadenas de caracteres — texto entre comillas, asociado a la cadena "asdfg".
  4. Operadores aritméticos — +, -, *, /.
  5. Operador de asignación — := (estilo Pascal).
  6. Operadores relacionales — >=, <=, >, <, =, <>.
  7. Símbolos delimitadores — { } [ ] ( ) , ; ..
  8. Palabras reservadas — if, else, for, print, int, más todas las permutaciones de "asdfg".

Interpretación del enunciado

Sobre las "combinaciones posibles con la cadena asdfg" como palabras reservadas: interpreté que se refiere a todas las permutaciones de las 5 letras {a, s, d, f, g}. Matemáticamente son 5! = 120 combinaciones únicas (asdfg, asdgf, adsfg, ..., gfsda, etc.), todas tratadas como palabras reservadas del lenguaje.

Sobre "leer un código fuente escrito en cualquier lenguaje de alto nivel": el analizador acepta cualquier archivo de texto. Clasifica lo que reconoce según las reglas definidas y reporta como Error todo lo que no pertenece al lenguaje (caracteres especiales no definidos, identificadores demasiado largos, números fuera de rango).

El analizador no detiene el análisis al encontrar un error. Continúa procesando el resto del código, permitiendo detectar múltiples errores en una sola pasada — comportamiento estándar en compiladores reales.

Diseño de la Solución

Estrategia: máquina de estados

El analizador implementa una máquina de estados que recorre el texto carácter por carácter. En cada posición decide qué tipo de token está comenzando según el carácter actual, delega a un método especializado de lectura, y agrega el token a la lista de resultados.

Decisiones de diseño importantes

DecisiónSolución adoptada
Palabras reservadas base Insensibles a mayúsculas: IF, If e if son equivalentes
Permutaciones de "asdfg" Sensibles a mayúsculas: solo minúsculas son palabras reservadas
Operadores de dos caracteres Se usa lookahead (mirar el siguiente carácter) para distinguir < de <= y <>
Comentarios Se ignoran (//, #, /* */) para soportar múltiples lenguajes fuente
Número fuera de rango Se reporta como Error con el valor encontrado para facilitar la depuración
Cadenas sin cerrar Se reporta Error si la cadena llega a fin de línea sin comilla de cierre
Seguimiento de posición Cada token guarda número de línea, posición inicial y final en el texto

Las 120 palabras reservadas de "asdfg"

Se generan en tiempo de ejecución calculando todas las permutaciones del conjunto de letras {a, s, d, f, g} usando un algoritmo recursivo: 

function* permutations(arr) {
  if (arr.length <= 1) { yield arr; return }
  for (let i = 0; i < arr.length; i++) {
    const rest = [...arr.slice(0, i), ...arr.slice(i + 1)]
    for (const perm of permutations(rest)) yield [arr[i], ...perm]
  }
}

// Genera: asdfg, asdgf, adsfg, adgfs, ... (120 en total)
const ASDFG_PERMS = new Set(
  [...permutations([...'asdfg'])].map(p => p.join(''))
)

¿Por qué React + Vite?

El enunciado permite usar cualquier herramienta de desarrollo. Se eligió React + Vite porque permiten construir una interfaz visual interactiva que va más allá de imprimir tokens en consola:

  • El código fuente se resalta en colores según el tipo de cada token
  • Los tokens se pueden filtrar por categoría en tiempo real
  • El hover entre código y lista de tokens es bidireccional
  • Se puede acceder desde cualquier navegador sin instalar nada
La lógica del analizador léxico (lexer.js) está completamente separada de la interfaz. Es JavaScript puro que podría usarse en cualquier entorno: Node.js, navegador, o como módulo en otro proyecto.

Explicación del Código

6.1 — Estructura del analizador (lexer.js)

El analizador es una clase con tres grupos de métodos:

class AnalizadorLexico {
  constructor(source) {
    this.src    = source  // código fuente completo
    this.pos    = 0       // posición actual en el texto
    this.line   = 1       // número de línea (para reportes)
    this.tokens = []      // lista de tokens encontrados
  }

  // Grupo 1: acceso al texto (sin modificar estado)
  _ch()    { ... }   // carácter actual
  _peek() { ... }   // mirar adelante (lookahead)
  _advance() { ... } // consumir carácter y avanzar

  // Grupo 2: saltear texto sin significado
  _skipBlanks() { ... }  // espacios + comentarios

  // Grupo 3: leer un token específico
  _readWord()   { ... }  // identificadores y reservadas
  _readNumber() { ... }  // números enteros
  _readString() { ... }  // cadenas entre comillas

  // Método principal
  analyze() { ... }      // recorre el fuente y produce tokens
}

6.2 — Lookahead para operadores de dos caracteres

Para distinguir <, <= y <> se usa el método _peek() que "mira" el siguiente carácter sin consumirlo: 

if (ch === '<') {
  const sig = this._peek()
  if      (sig === '=')  { /* consume dos: <= */ }
  else if (sig === '>')  { /* consume dos: <> */ }
  else                   { /* consume uno:  <  */ }
}

6.3 — Validación de identificadores

_readWord(startPos) {
  let word = ''
  while (letra_o_digito(this._ch())) word += this._advance()

  if (word.length > 10)
    return this._add(TK.ERROR, `identificador demasiado largo: "${word}"`, startPos)

  if (BASE_RESERVED.has(word.toLowerCase()))
    return this._add(TK.RESERVED, word.toLowerCase(), startPos)

  if (ASDFG_PERMS.has(word))
    return this._add(TK.RESERVED, word, startPos)

  this._add(TK.IDENTIFIER, word, startPos)
}

6.4 — Resaltado del código fuente

La función buildSegments() convierte la lista de tokens en segmentos del texto, que React luego renderiza como <span> con colores según el tipo de token: 

export function buildSegments(source, tokens) {
  const sorted = [...tokens].sort((a, b) => a.startPos - b.startPos)
  let last = 0, segments = []

  for (const tok of sorted) {
    if (tok.startPos > last)  // texto entre tokens (espacios)
      segments.push({ text: source.slice(last, tok.startPos), token: null })
    segments.push({ text: source.slice(tok.startPos, tok.endPos), token: tok })
    last = tok.endPos
  }
  return segments
}

Conclusiones

El analizador léxico desarrollado cumple con todos los requerimientos del enunciado: reconoce los 9 tipos de token definidos, reporta errores léxicos con detalle, procesa código fuente de cualquier lenguaje de alto nivel y produce un listado completo de los tokens detectados.

Ir más allá de la salida en consola y construir una interfaz visual permitió entender mejor el funcionamiento del analizador: ver el código fuente coloreado en tiempo real hace evidente cómo el lexer "ve" el texto — no como líneas de código, sino como una secuencia de categorías semánticas.

Algunos aspectos relevantes del aprendizaje:

  • El uso de lookahead es esencial para operadores de dos caracteres como :=, <=, <> y ..
  • Las 120 palabras reservadas generadas por permutación demuestran que el vocabulario de un lenguaje puede ser matemáticamente extenso, aunque en la práctica la mayoría de los lenguajes tienen menos de 50 palabras reservadas
  • Separar la lógica del analizador de la interfaz gráfica es una práctica de ingeniería importante: el archivo lexer.js podría usarse como módulo en la siguiente fase (analizador sintáctico) sin cambiar nada
  • El deploy continuo con GitHub Actions es una herramienta práctica que todo proyecto de software debería tener desde el inicio
El proyecto está disponible públicamente en aleosorio22.github.io/analizador-lexico y puede ser inspeccionado, probado y extendido para la siguiente etapa de la materia.
